# BQ-safe

> 一个正在持续迭代的资产安全检测工具集。当前已实现 URL 资产存活检测，后续会逐步扩展指纹识别、漏洞分析、子域名发现、隐藏路径收集等能力，目标是做成一个覆盖"资产发现 → 存活确认 → 指纹识别 → 风险分析"全流程的轻量级安全工具。

---

## ✨ 项目定位

BQ-safe 不只是一个URL存活检测脚本，而是按模块化方式在搭建一整套资产安全检测工具集。规划中的能力包括：

- ✅ **URL资产存活检测**（已实现）：`requests`初筛 + `Selenium`渲染复核，区分真实存活/需人工复核/确认死亡
- 🚧 **指纹识别**（规划中）：识别存活资产的技术栈、组件、后台框架等特征
- 🚧 **子域名发现**（规划中）：从给定根域名出发，枚举收集关联子域名资产
- 🚧 **隐藏路径收集**（规划中）：对存活站点做常见路径/接口的字典探测
- 🚧 **漏洞分析**（规划中）：结合指纹信息识别已知风险点，辅助人工判断

各模块会先独立开发验证，再逐步整合进统一的扫描流程和报告输出。目前仓库里的代码对应的是**第一个模块（URL存活检测）**，其余部分见下方 Roadmap。

## 🧱 当前模块：URL 存活检测 —— 工作流程

```
输入 .txt 文件（每行一个URL）
        │
        ▼
   去重 + 补全协议头（自动识别裸域名，补上 http://）
        │
        ▼
┌───────────────────────────────┐
│  阶段一：requests 多线程初筛     │
│  状态码 < 400        → alive   │
│  401 / 403 / 405 等  → possible│
│  404                 → dead    │
│  其余状态码 / 请求异常 → possible / dead │
└───────────────────────────────┘
        │  排除已确认 dead 的URL
        ▼
┌───────────────────────────────┐
│  阶段二：Selenium 渲染复核       │
│  检查 title + 页面内容特征      │
│  真实有内容页面      → alive   │
│  空壳 / 默认页 / 报错页 → dead  │
│  渲染异常（超时等）   → possible│
└───────────────────────────────┘
        │
        ▼
两阶段结果合并（以Selenium复核结果为准，没复核到的用request结果兜底）
按 alive / possible / dead 分类，写入 outputs/<时间戳>/result.json
```

## 🚀 快速开始（URL存活检测）

### 环境依赖

```bash
pip install requests selenium urllib3
```

需要本地装有 Chrome 浏览器（`Selenium` 4.6+ 会自动匹配对应版本的驱动，无需手动下载 chromedriver）。

### 准备输入文件

新建一个 `.txt` 文件，每行一个URL（裸域名也可以，会自动补全为 `http://`）：

```text
https://example.com
https://api.example.com/v1/user
sub.example.com
```

### 运行

```bash
python main.py -f urls.txt
```

运行后会先打印本次配置，并要求二次确认后才真正开始扫描：

```
输入文件：urls.txt
输出文件：/xxx/BQ-safe/outputs
request请求线程数：5
selenium请求线程：1
确认开始扫描?(y/n):
```

### 参数说明

| 参数 | 说明 | 默认值 |
|---|---|---|
| `-f`, `--file` | **必填**，待检测URL列表文件路径 | - |
| `-o`, `--output` | 输出文件名（仅文件名，不支持自定义目录），支持 `.json` / `.txt` | `result.json` |
| `-tr`, `--request_threads` | `requests` 阶段并发线程数 | 5 |
| `-ts`, `--selenium_threads` | `Selenium` 阶段并发线程数 | 1 |

> `Selenium` 阶段是真实起浏览器渲染，比纯HTTP请求重得多，线程数不建议设太高，否则本机资源容易吃紧。

### 输出结果

结果按时间戳保存在 `outputs/<年-月-日_时-分-秒>/` 目录下，结构大致如下：

```json
{
  "alive": [
    { "url": "https://example.com", "status": 200, "alive": "alive", "message": "success" }
  ],
  "possible": [
    { "url": "https://api.example.com", "title": null, "alive": "possible", "message": "渲染超时" }
  ],
  "dead": [
    { "url": "https://old.example.com", "status": 404, "alive": "dead", "message": "success" }
  ]
}
```

`possible` 档位建议人工复查一遍——通常是权限拦截（401/403）、服务端瞬时错误（5xx）、或者渲染超时这类"说不准"的情况。

## ⚙️ 配置项（`config.py`）

| 变量 | 说明 | 默认值 |
|---|---|---|
| `REQUEST_THREADS` | requests阶段默认线程数 | 5 |
| `SELENIUM_THREADS` | selenium阶段默认线程数 | 1 |
| `TIMEOUT` | 单个请求超时时间（秒） | 5 |
| `OUTPUT_DIR` | 结果输出根目录 | `<项目根目录>/outputs` |

## 🗺️ Roadmap

- [ ] 请求阶段增加重试机制
- [ ] 各模块整合为统一扫描流程与报告输出
